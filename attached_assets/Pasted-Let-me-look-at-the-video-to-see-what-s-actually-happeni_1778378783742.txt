**Bug: API requests are being cancelled when navigating between pages, causing UI to show empty/zero state**

**What's happening**
When clicking between tabs (CRM → Email → WhatsApp → Rates), in-flight API requests to the backend get cancelled instead of completing. This causes:
- Tab counters all flip to `0` (CRM 18 → 0, Email 13 → 0, WhatsApp 5 → 0, Quotes 6 → 0)
- Rates page shows "No rates yet" even though 137 rates exist
- Email inbox shows "Loading inbox..." indefinitely until a request finally succeeds
- The data IS still in the database — it eventually re-renders correctly (around 17 seconds in the recording, the inbox repopulates with all 13 emails)

**Network panel evidence** (filter: Errors only)
Every backend XHR shows status `(cancelled)`:
- `POST /re-extract` → cancelled (after 2266ms)
- `GET /rfqs` → cancelled (after 3ms)
- `POST /re-extract` → cancelled (after 468ms)
- `GET /rfqs` → cancelled (after 0ms)

All targeting `oneport-365-rfq-intake.re...`

**Root cause (likely)**
This is a classic React data-fetching bug. One of these is happening:
1. A `useEffect` cleanup function is calling `AbortController.abort()` on every re-render or route change, killing requests before they resolve.
2. React Query / SWR / TanStack Query is being initialized with a too-aggressive `signal` abort, or the query keys are changing on every render causing refetch+cancel loops.
3. The component fetching data unmounts (route change) before the fetch resolves, and there's no caching layer to retain the data.

**What to check / fix**
1. Look at the data-fetching hook(s) for the tab counts, rates list, and inbox. Check if `AbortController` is being aborted in a `useEffect` cleanup unconditionally.
2. If using React Query / SWR, ensure `staleTime` and `cacheTime` are set so cached data renders instantly while a refetch happens in the background — empty state should never show when cached data exists.
3. The empty-state UI (`No rates yet`, counter `0`) should only render when the query has **successfully returned an empty array**, not when it's in `loading` or `cancelled` state. Add an explicit check: `if (isLoading || isFetching) return <Spinner/>` before the empty state.
4. Confirm fonts on `unpkg.com` (4 woff2 files showing CACHE_MISS) aren't a real problem — those are just cache misses, not failures, and probably unrelated.
